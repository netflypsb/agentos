### The 6 Components (Refined from Research)

Each component is an MCP **tool** exposed through the single server:

#### 1. 📁 `format_convert` — Universal File Transformer
> *"The hardest part of writing a report is rarely the writing"* — Anthropic on Claude Cowork

| Feature | Details |
|---|---|
| What | Convert between file formats locally. No external APIs. |
| Conversions | MD↔HTML, CSV↔JSON, YAML↔JSON, XML→JSON, plaintext→structured MD |
| Why agents need it | Claude Cowork constantly reads/writes files. OpenClaw processes messages across channels. Every agent needs to normalize input formats. |
| Why it's a pick-axe | Agents call this hundreds of times per workflow. It's plumbing. |
| Implementation | Pure library calls (Pandoc-like logic, built-in parsers). No network. |

**Excluded** (compliance risk): PDF generation (font licensing), DOCX (MS format patents edge cases), OCR (image processing → potential PII exposure)

#### 2. 📋 `task_ledger` — Persistent Task State & Checkpoints
> *"Step 3 picked the wrong tool. Steps 4–12 were doomed. If your eval only checks the final output, you'll never know why."* — AI Agents Stack 2026

| Feature | Details |
|---|---|
| What | SQLite-backed task state machine with checkpoints, rollback, and audit trail |
| Tools | `task_create`, `task_checkpoint`, `task_rollback`, `task_status`, `task_list`, `task_complete` |
| Why agents need it | Multi-step workflows crash, time out, or go wrong at step 7 of 12. Without state persistence, you restart from zero. |
| Why it's a pick-axe | Every orchestration framework (LangGraph, CrewAI, custom) needs durable state. This is the missing layer. |
| Implementation | SQLite file per workspace. JSON state blobs. No external DB. |

#### 3. 📊 `agent_log` — Structured Agent Activity Logger
> *"89% of organizations have observability, but only 52% have evals. That 37-point gap is where production quality dies."* — LangChain State of Agent Engineering

| Feature | Details |
|---|---|
| What | Append-only structured log of every agent action: tool calls, decisions, inputs/outputs, timing, token estimates |
| Tools | `log_action`, `log_search`, `log_summary`, `log_export` |
| Why agents need it | Debugging agent behavior requires seeing the full reasoning chain. Dashboards show the fire; this records how it started. |
| Why it's a pick-axe | Required for every production agent. Currently everyone builds this ad-hoc. |
| Implementation | JSONL append-only file. SQLite index for search. Zero network calls. |

#### 4. ⏱️ `budget_guard` — Pre-Execution Cost & Loop Protection
> *"A $50 POC becomes an $847,000 monthly production bill."* — Developer community analysis

| Feature | Details |
|---|---|
| What | Token/action budget tracking per task. Loop detection. Automatic stop before limits are exceeded. |
| Tools | `budget_set`, `budget_check`, `budget_consume`, `budget_status`, `loop_detect` |
| How loop detection works | Tracks action fingerprints (tool name + truncated params hash). If the same fingerprint appears N times in M seconds, it flags a loop. |
| Why agents need it | Agents can't self-limit. They don't know they've been running for 3 hours and burned $8K. This says "no" *before* the expensive call. |
| Why it's a pick-axe | The #1 production pain point cited across Reddit, HN, and industry reports. |
| Implementation | In-memory counters backed by SQLite. No external service. |

#### 5. 📝 `template_engine` — Boilerplate & Document Scaffolding
> *"Organizing and managing local files... Preparing documents from source files"* — Top Claude Cowork use cases

| Feature | Details |
|---|---|
| What | Templating engine with variable injection, conditional sections, and repeating blocks |
| Tools | `template_list`, `template_render`, `template_create` |
| Built-in templates | Meeting notes, project status report, comparison table, pros/cons analysis, weekly summary, changelog |
| Why agents need it | Agents waste tokens regenerating the same document structures. Templates are cheaper than re-prompting. |
| Why it's a pick-axe | Reduces token cost per document by 40-60%. Every knowledge worker agent needs this. |
| Implementation | Jinja2-style template engine. Templates stored as local files. |

#### 6. 🔑 `kv_store` — Fast Local Key-Value Memory
> *"Memory became a first-class architectural primitive"* — AI Agents Stack 2026

| Feature | Details |
|---|---|
| What | Persistent key-value store with TTL, namespaces, and prefix search |
| Tools | `kv_set`, `kv_get`, `kv_delete`, `kv_list`, `kv_search` |
| Why agents need it | Agents need to remember things between sessions: user preferences, previous results, cached computations, workspace state. Current options are either full vector DBs (overkill) or nothing. |
| Why it's a pick-axe | The simplest useful memory layer. No embeddings, no vectors, no ML — just persistent data. |
| Implementation | SQLite with automatic namespacing. TTL via background cleanup. |
